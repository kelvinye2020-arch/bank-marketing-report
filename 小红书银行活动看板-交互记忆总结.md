# 小红书银行营销活动看板 — 完整交互记忆

## 一、项目概述

这是一个**小红书银行营销活动搜索看板**，通过小红书 MCP（Model Context Protocol）自动搜索银行相关的营销活动笔记，生成一个可视化的 HTML 报告页面，部署在 GitHub Pages 上供浏览。

**项目仓库**：https://github.com/kelvinye2020-arch/bank-marketing-report（推测，实际在 `c:/Users/kelvinyye/WorkBuddy/20260313150001/`）

**核心文件**：
- `generate_report.py` — 读取搜索结果 JSON，过滤、排序、生成 HTML 报告
- `search_result_1~4.json` — 小红书 MCP 搜索返回的原始数据
- `bank_marketing_report.html` — 生成的最终报告页面

### generate_report.py 核心逻辑

2. **银行相关过滤**：标题中包含银行/立减/信用卡/满减等关键词

## 六、踩过的坑与经验教训

1. **百度 `site:xiaohongshu.com` 搜不到内容**：百度对小红书收录极差，不要用 `site:` 限定搜小红书内容
2. **小红书 explore 链接需要登录**：`xiaohongshu.com/explore/{note_id}` 在浏览器打开会跳转到登录页，不适合作为公开链接
3. **小红书搜索链接可以免登录使用**：`xiaohongshu.com/search_result?keyword=xxx` 可以在未登录状态下使用，是当前最佳方案
4. **EdgeOne Pages CDN 缓存**：更新后可能需要等几分钟或强制刷新才能看到最新内容
5. **PowerShell 引号问题**：在 PowerShell 中 `git commit -m` 后面的中文引号会出问题，应使用简单的英文字符串
6. **用户对 git 操作的偏好**：用户不喜欢 `git reset --hard` 等破坏性操作，应使用更温和的方式（如检出文件覆盖）
7. **小红书笔记正文反爬极强**：`get_feed_detail` 接口有 x-s 动态签名 + xsec_token 时效性 + 频率限制（连续 3-5 次即封）+ 浏览器指纹检测。MCP 批量抓取在无代理池条件下不可行（实测 13 条只成功 1 条）。
8. **⚠️ 改造纪律：先小规模测试，再全量执行**：依赖外部接口的改造，必须先拿 2-3 条测试可行性，确认通了再全量跑。不要一上来就全量调用，浪费资源后才发现路不通。

- **简洁优先**：宁可功能少但稳定好用，不要花哨但不稳定的功能

- **改造纪律**：大改造先小规模测试可行性，不要一上来全量跑

### 反爬限制实测结论（2026-04-12）

| 接口 | 反爬强度 | 说明 |
|------|---------|------|
| `search_feeds` | 🟢 **宽松** | 搜索 4-5 组关键词无问题，是看板数据的可靠来源 |
| `get_feed_detail` | 🔴 **极强** | 连续 3-5 次即被封锁，需代理池+频率控制+token 刷新，轻量项目不可行 |

### 四层反爬体系
1. **x-s / x-t 动态签名** — MCP 的 Playwright 底层已处理，搜索能通过
2. **xsec_token 时效性** — 从搜索结果获取的 token 有效期短，批量抓详情时后面的 token 可能已过期
3. **请求频率智能限制** — 连续请求 3-5 次后被封，单次间隔要 ≥3 秒，还需代理池轮换
4. **浏览器指纹检测** — MCP 底层走 Playwright，频率一高就被识别为自动化工具
